# akb — AI Knowledge Base Generator

**Date:** 2026-02-13
**Status:** Brainstorm

## What We're Building

A Go CLI tool (`akb`) that analyzes a source code repository and generates markdown documentation for every source file. Each markdown file describes the file's purpose and all its symbols (functions, methods, classes, structs, interfaces, etc.) in detail. The output is a knowledge base optimized for AI agents to use as context when implementing features, modifying code, or debugging.

### Core Workflow

1. **`akb init`** — Walks the repo, collects all unique file extensions/filenames, sends them to Claude CLI for classification (source code vs. not). Writes the result to a config file that users can review and edit.
2. **`akb generate`** — Reads the config, filters source files, checks a manifest for changes (content hashes), and shells out to Claude CLI concurrently to generate structured markdown for each changed file. Writes output to `docs/akb/` mirroring the source tree.

### Key Characteristics

- **LLM-powered analysis:** Uses Claude CLI (`claude -p`) as the analysis engine — no custom AST parsing needed initially.
- **Language-agnostic:** The LLM handles any programming language. No hardcoded language support.
- **Incremental:** Tracks file content hashes in a manifest to skip unchanged files on subsequent runs.
- **Concurrent:** Processes N files in parallel (configurable, default 5).
- **AI-agent-first output:** Markdown format optimized for machine consumption as context for coding assistants.

## Why This Approach

We chose the Two-Phase Pipeline (init + generate) because:

- **Clean separation:** Discovery (what to analyze) vs. analysis (how to describe it) are distinct concerns.
- **User control:** The config file lets users override the LLM's classification decisions.
- **Incremental by design:** The manifest naturally fits into the generate phase.
- **LLM cost efficiency:** Classification happens once (init), analysis only runs on changed files (generate).

Alternatives considered:
- **Single-pass (classify + analyze in one LLM call per file):** Simpler flow but re-classifies every run, no reusable config, more LLM calls for non-source files.
- **Heuristic extension list + LLM analysis:** Faster filtering but requires maintaining an extension list and misses unusual file types.

## Architecture

### CLI Interface

```
akb init [--path <repo-path>]
akb generate [--path <repo-path>] [--workers N] [--force]
```

| Flag | Description | Default |
|---|---|---|
| `--path` | Repository root path | Current directory |
| `--workers` | Number of concurrent Claude CLI calls | 5 |
| `--force` | Regenerate all files, ignoring manifest | false |

### Output Structure

```
repo/
├── src/foo.go
├── src/bar.py
├── lib/util.ts
└── docs/akb/
    ├── .config.yaml          # LLM-generated source extensions + user overrides
    ├── .manifest.json         # File content hashes for incremental detection
    ├── src/
    │   ├── foo.go.md          # Knowledge base for foo.go
    │   └── bar.py.md          # Knowledge base for bar.py
    └── lib/
        └── util.ts.md         # Knowledge base for util.ts
```

### Config File (`.config.yaml`)

```yaml
# Auto-generated by akb init. Edit as needed.
source_extensions:
  - .go
  - .py
  - .ts
  - .js
  - .rb
  - .rs
exclude_patterns:
  - vendor/
  - node_modules/
  - .git/
```

### Manifest File (`.manifest.json`)

```json
{
  "src/foo.go": "sha256:abc123...",
  "src/bar.py": "sha256:def456..."
}
```

### Incremental Strategy

1. Walk source files matching config extensions, excluding patterns
2. Compute SHA-256 content hash for each file
3. Compare with manifest — skip unchanged files
4. Process changed/new files via Claude CLI
5. Remove stale markdown for deleted source files
6. Update manifest

### Claude CLI Integration

**Init classification prompt:** Given a list of file extensions/filenames, return JSON classifying which are source code.

**Analysis prompt:** Given a source file's content and path, generate structured markdown with:
- File-level summary (purpose, dependencies)
- For each symbol: name, signature, visibility, description, parameters, return values, notable side effects

Invocation: `claude -p "<prompt>" --output-format text`

### Error Handling

- Claude CLI not found → clear error with install instructions
- Claude CLI fails for a specific file → log error, continue processing, report failures at end
- No config when running `generate` → suggest running `init` first

## Key Decisions

1. **Go as the implementation language** — matches the project setup, good concurrency primitives.
2. **Shell out to Claude CLI** — simple, leverages existing installation, avoids API key management.
3. **LLM-powered language detection** — no hardcoded extension lists, handles any language.
4. **Two-phase pipeline (init + generate)** — clean separation, user-editable config, cost-efficient.
5. **Mirror source tree for output** — intuitive navigation, easy for AI agents to locate documentation.
6. **SHA-256 content hashes for incremental** — reliable change detection, stored in manifest.
7. **Configurable concurrency** — balance speed vs. API cost, default 5 workers.
