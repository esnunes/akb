# feat: akb CLI — AI Knowledge Base Generator

## Overview

A Go CLI tool that analyzes source code repositories and generates markdown knowledge base files optimized for AI agent consumption. Uses Claude CLI as an external subprocess for LLM-powered analysis. Two subcommands: `init` (discover source file types) and `generate` (produce markdown documentation).

## Problem Statement / Motivation

AI coding agents (Claude Code, Copilot, Cursor) need codebase context to implement features, fix bugs, and understand code. Currently there is no automated way to produce a structured knowledge base of a repository's symbols (functions, methods, classes, structs) that agents can consume as context. `akb` fills this gap.

## Proposed Solution

Two-phase pipeline:

1. **`akb init`** — Walks the repo, collects unique file extensions, asks Claude CLI to classify which are source code, writes a config file the user can review/edit.
2. **`akb generate`** — Reads config, walks source files, checks a manifest for changes (SHA-256 hashes), shells out to Claude CLI concurrently to generate structured markdown per file, writes output mirroring the source tree.

## Technical Approach

### Project Structure

```
akb/
├── main.go                    # Entry point, subcommand dispatch
├── internal/
│   ├── config/
│   │   └── config.go          # Load/save .config.yaml
│   ├── manifest/
│   │   └── manifest.go        # Load/save .manifest.json, hash comparison
│   ├── walker/
│   │   └── walker.go          # Walk repo, collect extensions, filter files
│   ├── claude/
│   │   └── claude.go          # Shell out to claude CLI
│   └── analyzer/
│       └── analyzer.go        # Concurrent file analysis orchestration
├── go.mod
├── go.sum
└── docs/
```

### CLI Interface

Single binary with subcommand dispatch via `os.Args[1]`:

```
akb init [--path <repo-path>]
akb generate [--path <repo-path>] [--workers N] [--force]
akb help
```

| Flag | Default | Description |
|---|---|---|
| `--path` | `.` | Repository root path (resolved relative to CWD) |
| `--workers` | `5` | Concurrent Claude CLI calls (min 1, max 20) |
| `--force` | `false` | Regenerate all files, ignore manifest |

Use stdlib `flag` package with `flag.NewFlagSet` per subcommand.

### Output Structure

All output goes to `docs/akb/` inside the target repo (the `--path` directory):

```
<repo>/docs/akb/
├── .config.yaml          # Source extensions + exclude patterns
├── .manifest.json        # File content hashes
├── src/
│   ├── foo.go.md         # Knowledge base for foo.go
│   └── bar.py.md         # Knowledge base for bar.py
└── lib/
    └── util.ts.md
```

### Config File (.config.yaml)

```yaml
# Generated by akb init. Edit as needed.
source_extensions:
  - .go
  - .py
  - .ts
exclude_patterns:
  - .git/
  - node_modules/
  - vendor/
  - docs/akb/
```

**Init re-run behavior:** If config already exists, print a warning and skip. User must delete the config to regenerate it, or edit it manually.

### Manifest File (.manifest.json)

```json
{
  "src/foo.go": "sha256:abc123def456...",
  "src/bar.py": "sha256:789abc012def..."
}
```

Hashes are SHA-256 of file content only (not config state). Written atomically via temp file + rename.

### Claude CLI Integration

#### Pre-checks

Before any Claude CLI call, verify `claude` is in PATH using `exec.LookPath`. If missing, error with install instructions.

#### Classification Prompt (init)

Send to `claude -p <prompt> --output-format json`:

```
Given the following list of file extensions found in a source code repository,
return a JSON object with two arrays:

1. "source_extensions": extensions that are programming language source code files
   (code that defines functions, classes, methods, structs, etc.)
2. "non_source_extensions": everything else (configs, data, docs, binaries, etc.)

Be inclusive — if an extension could be source code, include it in source_extensions.

Extensions found:
<list of extensions>
```

Expected response format:
```json
{
  "source_extensions": [".go", ".py", ".ts"],
  "non_source_extensions": [".json", ".md", ".yaml", ".png"]
}
```

Parse with `encoding/json`. If parsing fails, log the raw response and error with a clear message.

#### Analysis Prompt (generate)

Send to `claude -p <prompt> --output-format text`:

```
Analyze the following source code file and generate a structured markdown
knowledge base entry. This will be used as context by AI coding agents.

File path: <relative path>

For each symbol (function, method, class, struct, interface, type, constant,
variable) in the file, document:
- Name and signature
- Brief description of what it does
- Parameters and return values (if applicable)
- Notable dependencies, side effects, or relationships

Start with a brief file-level summary (1-2 sentences).

Use this exact format:

# <filename>

<file summary>

## Functions

### <function_name>

`<signature>`

<description>

## Types

### <type_name>

`<definition>`

<description>

(Use appropriate sections: Functions, Types, Methods, Constants, Variables, etc.
Omit empty sections.)

Source code:
```
<file content>
```
```

Pass file content via stdin to avoid shell escaping issues:

```go
cmd := exec.CommandContext(ctx, "claude", "-p", prompt, "--output-format", "text")
cmd.Stdin = strings.NewReader(sourceCode)
```

Wait — Claude CLI's `-p` flag reads the prompt as an argument. Instead, embed the file content in the prompt and pipe the full prompt via a temp file or use `--` conventions. The simplest approach: build the full prompt string (including the source code) and pass it as the `-p` argument. For very large files, this could hit argument length limits, so use stdin:

```go
cmd := exec.CommandContext(ctx, "claude", "-p", prompt)
output, err := cmd.Output()
```

Where `prompt` includes the full source code inline.

#### Timeout

Each Claude CLI call gets a 120-second context timeout. If exceeded, the file is logged as failed and the worker moves on.

#### Rate Limits and Retries

If Claude CLI exits with a non-zero code, retry up to 2 times with exponential backoff (2s, 4s). After 3 total attempts, log the failure and continue.

### Incremental Strategy

1. Load manifest (or start empty if none exists)
2. Walk source files matching config
3. For each file, compute SHA-256 hash
4. Skip if hash matches manifest (unless `--force`)
5. Process changed/new files via worker pool
6. Detect stale entries: manifest keys not found in current file walk → delete corresponding `.md` files
7. Write updated manifest atomically (temp file + rename)

`--force` skips step 4 (processes all files) but still runs step 6 (stale cleanup).

### Concurrency

Worker pool pattern with a buffered channel as semaphore:

```go
sem := make(chan struct{}, workers)
var wg sync.WaitGroup
var mu sync.Mutex
var errors []error

for _, file := range changedFiles {
    wg.Add(1)
    sem <- struct{}{} // acquire
    go func(f FileInfo) {
        defer wg.Done()
        defer func() { <-sem }() // release
        if err := processFile(ctx, f); err != nil {
            mu.Lock()
            errors = append(errors, err)
            mu.Unlock()
        }
    }(file)
}
wg.Wait()
```

### File Walking

Use `filepath.WalkDir` (more efficient than `filepath.Walk`). Skip:
- Directories matching `exclude_patterns` from config
- Files whose extension is not in `source_extensions`
- The `docs/akb/` output directory itself (hardcoded exclusion)
- Symlinks (do not follow — avoids cycles)

### Progress Output

Print to stderr:
```
akb: init — classifying 23 unique extensions...
akb: init — wrote docs/akb/.config.yaml (15 source extensions)

akb: generate — scanning files...
akb: generate — 42 files to process (18 changed, 24 cached)
akb: generate — [1/18] src/foo.go
akb: generate — [2/18] src/bar.py
...
akb: generate — done (18 processed, 0 failed, 24 cached)
```

### Error Handling

| Scenario | Behavior | Exit Code |
|---|---|---|
| Claude CLI not in PATH | Error with install URL | 1 |
| Config not found (generate) | Error: "run akb init first" | 1 |
| Config already exists (init) | Warning, skip init | 0 |
| All files processed OK | Success | 0 |
| Some files failed | Log failures, continue | 1 |
| All files failed | Error summary | 1 |
| Invalid --workers value | Error with valid range | 1 |

### Security Considerations

- Use `exec.Command` (not `exec.CommandContext` with shell) to avoid command injection — Go's `exec.Command` does not invoke a shell.
- File content is passed as part of the prompt string argument, not interpolated into a shell command.
- The tool sends source code to Claude's API via the CLI — this is inherent to the design. No secrets detection is built in (users should manage this via `.gitignore` and `exclude_patterns`).

## Implementation Phases

### Phase 1: Project Scaffolding and CLI Framework

Set up the Go project structure, subcommand dispatch, and flag parsing.

**Files:**
- `main.go` — entry point, subcommand routing
- `internal/config/config.go` — Config struct, Load/Save functions
- `internal/manifest/manifest.go` — Manifest struct, Load/Save/Hash functions

**Tasks:**
1. Create `main.go` with subcommand dispatch (`init`, `generate`, `help`)
2. Implement flag parsing with `flag.NewFlagSet` per subcommand
3. Implement config load/save (YAML) with validation
4. Implement manifest load/save (JSON) with atomic writes
5. Implement SHA-256 file hashing

### Phase 2: File Walking and Filtering

Implement the repo walking logic used by both `init` and `generate`.

**Files:**
- `internal/walker/walker.go` — DiscoverExtensions, WalkSourceFiles

**Tasks:**
1. Implement `DiscoverExtensions(path)` — walks repo, returns unique extensions
2. Implement `WalkSourceFiles(path, config)` — walks repo, returns files matching config
3. Handle exclude patterns and hardcoded exclusions (`.git/`, `docs/akb/`)
4. Skip symlinks

### Phase 3: Claude CLI Integration

Implement the subprocess wrapper for calling Claude CLI.

**Files:**
- `internal/claude/claude.go` — ClassifyExtensions, AnalyzeFile

**Tasks:**
1. Implement pre-check (`exec.LookPath("claude")`)
2. Implement `ClassifyExtensions(ctx, extensions)` — sends classification prompt, parses JSON response
3. Implement `AnalyzeFile(ctx, path, content)` — sends analysis prompt, returns markdown string
4. Add 120s timeout per call
5. Add retry logic (2 retries, exponential backoff)

### Phase 4: Init Command

Wire up the init flow.

**Files:**
- Update `main.go` init handler

**Tasks:**
1. Check if config exists → warn and skip if so
2. Call `walker.DiscoverExtensions`
3. Call `claude.ClassifyExtensions`
4. Add default exclude patterns (`.git/`, `node_modules/`, `vendor/`, `docs/akb/`)
5. Save config
6. Print summary to stderr

### Phase 5: Generate Command

Wire up the generate flow with concurrent processing.

**Files:**
- `internal/analyzer/analyzer.go` — orchestrates concurrent analysis
- Update `main.go` generate handler

**Tasks:**
1. Load config (error if missing)
2. Walk source files
3. Load manifest
4. Compute hashes, determine changed files
5. Implement worker pool with semaphore pattern
6. For each changed file: call `claude.AnalyzeFile`, write markdown, update manifest entry
7. Detect and remove stale markdown files
8. Write manifest atomically
9. Print progress and summary to stderr
10. Set exit code based on success/failure

### Phase 6: Testing and Polish

**Tasks:**
1. Add table-driven tests for config load/save
2. Add tests for manifest load/save/hash comparison
3. Add tests for walker (extension discovery, file filtering)
4. Add tests for Claude CLI wrapper (mock `exec.Command`)
5. Manual end-to-end test against a real repository
6. Validate error messages and edge cases
